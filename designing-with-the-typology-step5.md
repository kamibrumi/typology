## Step 5: Reflection
When designing with the typology, ask:

- Which decisions are well supported?
- Which decisions are currently implicit in the system?
- Which decisions are automated?
- Which decisions require human judgment?
- Where does the user need comparison?
- Where does the user need control over criteria?
- Where does the user need explanation?
- Where does the user need to create or revise information?
- Where should the interface support iteration?

## Satellite Example
When designing the satellite decision-support tool, ask:

- What is the final decision stakeholders need to make?
- What candidate designs already exist?
- Where do users need to create new designs or new attributes?
- Which constraints are hard requirements?
- Which criteria are negotiable preferences?
- Which designs are removed by each constraint?
- Where do stakeholders need to compare tradeoffs?
- Where should uncertainty from simulations be shown?
- How should stakeholder disagreement be represented?
- Where should the interface support revising criteria or regenerating options?
- What evidence should be saved to justify the final decision?

## Summary
The collaborative satellite design example illustrates how the typology can guide visualization design after problem formulation.

The top-level decision is to **choose** a final satellite design. However, that choice depends on earlier **create** tasks, such as generating designs and estimating performance, and **activate** tasks, such as filtering designs by constraints and stakeholder acceptability.

By representing these tasks in a decision diagram, designers can identify what kinds of interface support are needed: generators, filters, simulation views, tradeoff visualizations, stakeholder satisfaction views, comparison tools, and final selection mechanisms.

This makes the typology useful not only for describing decision-making, but also for carrying the structure of a decision problem into visualization design.
